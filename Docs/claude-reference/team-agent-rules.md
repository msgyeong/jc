# 팀 에이전트 구성 & 품질 사이클

## 팀 구성

| 역할 | 설명 | MCP 도구 |
|------|------|----------|
| **사장** | 최종 의사결정자, 지시 및 방향 설정 (사용자) | — |
| **기획자 에이전트** | PRD, 유저 플로우, 요구사항 정의 (`Docs/`) | Sequential Thinking |
| **디자이너 에이전트** | UI/UX 디자인, 디자인 시스템 (`Docs/design-system/`) | Playwright (UI 검증) |
| **프론트엔드 에이전트** | `web/js/`, `web/styles/`, `web/index.html` 담당 | Playwright (동작 테스트) |
| **백엔드 에이전트** | `api/` (Express 라우트, 미들웨어, DB 쿼리) 담당 | Postgres (DB 직접 쿼리) |
| **테스터 에이전트** | 기능 테스트, API 테스트, 배포 검증 | Playwright + Postgres |

## MCP 서버 활용 규칙

### 1. Postgres MCP — 백엔드·테스터 필수 사용
- **코드 작성 전**: DB 스키마(테이블, 컬럼, 제약조건)를 Postgres MCP로 직접 확인
- **버그 수정 시**: 실제 DB 데이터를 쿼리해서 원인 확인 (추측 금지)
- **배포 후**: INSERT/SELECT로 기능이 정상 동작하는지 DB 레벨에서 검증
- **catch(_){} 금지**: DB 에러를 무시하지 말고, Postgres MCP로 즉시 원인 파악

### 2. Playwright MCP — 프론트엔드·디자이너·테스터 필수 사용
- **UI 변경 후**: Playwright로 실제 배포된 앱 화면을 열어서 확인
- **버그 재현**: 사장님이 보고한 버그를 Playwright로 직접 재현
- **네비게이션 테스트**: 화면 전환, 뒤로가기, 탭 전환을 자동으로 테스트
- **N뱃지/상태 테스트**: 게시글 읽기 → 뱃지 사라지는지 확인

### 3. Sequential Thinking MCP — 기획자·복잡한 버그 분석
- **복잡한 버그**: 여러 화면/기능이 얽힌 버그는 Sequential Thinking으로 원인 구조화
- **기능 설계**: 새 기능 추가 시 영향 범위 분석
- **리팩토링 계획**: 변경 순서, 의존성, 위험도 분석

## 팀 협업 필수 규칙

1. **문서 기준 작업**: `Docs/` 문서를 먼저 확인, 문서와 코드가 충돌하면 문서를 우선
2. **디자인 통일성 필수**: `Docs/design-system/` 가이드 준수, 독자적으로 새 스타일을 만들지 않음
3. **팀 단위 사고**: 프론트 변경 → 백엔드 API 확인, 백엔드 변경 → 프론트 호출부 확인
4. **작업 이력 기록**: `Docs/agent-logs/`에 로그 기록
5. **DB 확인 우선**: 코드만 보고 추측하지 말고, Postgres MCP로 실제 DB 상태 확인
6. **배포 후 검증**: Playwright로 실제 앱에서 동작 확인 후 완료 처리

## 품질 완성 사이클 — 한 번에 끝내기 (MCP 강화)

```
① 기획 확인  →  ② DB 스키마 확인  →  ③ 구현  →  ④ Playwright 검증  →  ⑤ 완료
```

| 단계 | 담당 | 행동 | MCP 활용 |
|------|------|------|----------|
| **① 기획 확인** | 기획자 | PRD·와이어프레임·정책 문서에서 요구사항 정리 | Sequential Thinking으로 복잡한 요구사항 구조화 |
| **② DB 확인** | 백엔드 | 테이블/컬럼이 코드와 일치하는지 확인 | **Postgres MCP로 직접 쿼리** |
| **③ 구현** | 디자이너 + 프론트 + 백엔드 | 디자인 시스템 준수, API 스펙 맞춤 확인 | — |
| **④ 검증** | 테스터 | API 호출 + UI 동작 검증, 기획·디자인과 비교 대조 | **Playwright로 실제 앱 테스트 + Postgres로 DB 검증** |
| **⑤ 완료** | 테스터 | 재검증 통과 확인 | Playwright 스크린샷으로 증거 확보 |

**핵심 원칙:**
- 테스터가 통과시키지 않은 기능은 완료가 아니다
- 문서와 코드가 다르면 문서 기준으로 코드를 수정
- 같은 버그로 사장님이 2번 이상 지적하는 일이 없도록 한 사이클에서 끝낸다
- **추측 금지**: DB 상태는 Postgres MCP로 확인, UI 상태는 Playwright로 확인
