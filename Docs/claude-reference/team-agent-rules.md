# 팀 에이전트 구성 & 품질 사이클

## 팀 구성

| 역할 | 설명 |
|------|------|
| **사장** | 최종 의사결정자, 지시 및 방향 설정 (사용자) |
| **기획자 에이전트** | PRD, 유저 플로우, 요구사항 정의 (`Docs/`) |
| **디자이너 에이전트** | UI/UX 디자인, 디자인 시스템 (`Docs/design-system/`) |
| **프론트엔드 에이전트** | `web/js/`, `web/styles/`, `web/index.html` 담당 |
| **백엔드 에이전트** | `api/` (Express 라우트, 미들웨어, DB 쿼리) 담당 |
| **테스터 에이전트** | 기능 테스트, API 테스트, 배포 검증 |

## 팀 협업 필수 규칙

1. **문서 기준 작업**: `Docs/` 문서를 먼저 확인, 문서와 코드가 충돌하면 문서를 우선
2. **디자인 통일성 필수**: `Docs/design-system/` 가이드 준수, 독자적으로 새 스타일을 만들지 않음
3. **팀 단위 사고**: 프론트 변경 → 백엔드 API 확인, 백엔드 변경 → 프론트 호출부 확인
4. **작업 이력 기록**: `Docs/agent-logs/`에 로그 기록

## 품질 완성 사이클 — 한 번에 끝내기

```
① 기획 확인  →  ② 구현  →  ③ 테스터 검증  →  ④ 비교/수정  →  ⑤ 완료
```

| 단계 | 담당 | 행동 |
|------|------|------|
| **① 기획 확인** | 기획자 | PRD·와이어프레임·정책 문서에서 요구사항 정리 |
| **② 구현** | 디자이너 + 프론트 + 백엔드 | 디자인 시스템 준수, API 스펙 맞춤 확인 |
| **③ 테스터 검증** | 테스터 | API 호출 + UI 동작 검증, 기획·디자인과 비교 대조 |
| **④ 비교/수정** | 전원 | 불일치·버그 즉시 수정 |
| **⑤ 완료** | 테스터 | 재검증 통과 확인 |

**핵심 원칙:**
- 테스터가 통과시키지 않은 기능은 완료가 아니다
- 문서와 코드가 다르면 문서 기준으로 코드를 수정
- 같은 버그로 사장님이 2번 이상 지적하는 일이 없도록 한 사이클에서 끝낸다
