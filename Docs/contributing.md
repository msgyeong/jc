# 영등포 JC 회원관리 앱 - 프로젝트 가이드라인

본 문서는 **영등포 JC 회원관리 앱** 프로젝트의 최우선 규칙을 정의합니다.

---

## 1. 개발 로그 관리 규칙

### 1.1 로그 폴더 구조
- 모든 개발 로그는 `log/` 폴더에 저장
- 로그 파일은 일자별로 관리

### 1.2 로그 파일 명명 규칙
- 파일명 형식: `YYMMDD.md` (예: `260108.md` - 2026년 1월 8일)
- 위치: `log/` 폴더 내

### 1.3 로그 작성 규칙
- 핵심 요점만 축약하여 기록
- 매일 개발 작업이 있을 때마다 해당 일자의 로그 파일에 기록
- 다음 항목 포함:
  - 작업 내용
  - 주요 변경 사항
  - 이슈 및 해결 방법 (있는 경우)
  - 다음 작업 계획 (있는 경우)

### 1.4 로그 작성 예시

```markdown
# 2026년 1월 8일 개발 로그

## 작업 내용
- PRD 문서 정리
- 앱 이름 '영등포 JC'로 통일

## 주요 변경 사항
- 참조 링크 전체 업데이트

## 다음 작업
- 프로젝트 가이드라인 파일 생성
- 로그 시스템 구축
```

## 2. 문서 관리 규칙

### 2.1 문서 관리
- 모든 기획/설계 문서는 `docs/` 폴더에서 관리

### 2.2 앱 이름 통일
- 공식 앱 이름: **영등포 JC** (공백 포함)
- 모든 문서에서 일관되게 사용

### 2.3 용어/표기 통일 (필수)
문서에서의 용어 혼동은 구현 단계에서 권한/역할 정의 오류를 유발한다.
따라서 아래 표준 용어를 **문서 전체에서 고정**한다.

- **관리자 웹**
  - 의미: 가입 승인/회원 관리/권한 부여를 수행하는 Web 콘솔
  - 금지 표기: `Web Admin`, `Admin Web`, `관리자 Web`, `관리자 web`, `관리자Web`
- **최고 관리자**
  - 의미: 관리자 웹의 최상위 권한 주체(예: `manager_###` 계정)
  - 금지 표기: `슈퍼 관리자`, `Super Admin` (문서에서는 사용하지 않음)
- **관리자 계정**
  - 의미: 최고 관리자가 관리자 웹에서 권한을 부여한 관리자 계정
  - 금지 표기: `Web Admin 계정` 등 역할명이 섞인 표현
- **앱 내 관리자(직책 기반)**
  - 의미: 앱에서 공지/일정 등 관리자 기능을 수행하는 회원(직책 기반 부여)
  - 금지 표기: `App Role Admin`, `앱 관리자 권한`(의미가 모호할 때)

참고: 코드(식별자/라이브러리 관례)에서는 영문 표기를 사용할 수 있으나,
**문서 상의 역할/권한 명칭은 위 표준 용어만 사용**한다.

### 2.4 개발 방향 변경 시 문서 우선 업데이트 원칙 (필수)
- **핵심 원칙**: 개발 방향이 수정되면 우선적으로 `docs/` 폴더 내의 개발/기획 문서를 업데이트하고, 그 후에 코딩을 진행
- 문서 업데이트 → 코드 변경 순서를 반드시 준수

#### 문서 수정 규칙
1. **절대 삭제 금지**: 수정/삭제되는 부분은 절대 완전히 삭제하지 않음
2. **수정/삭제 표시 방법**:
   - 삭제된 내용: 주석 처리 (예: `<!-- 삭제됨: 기존 내용 -->`) 또는 취소선 처리
   - 수정된 내용: 수정 전 내용은 주석 처리하고, 수정 후 내용을 명확히 표시
   - 변경 이력: 변경 날짜와 변경 이유를 함께 기록

#### 문서 수정 예시
```markdown
### 기존 내용 (수정 전)

<!-- [변경일: 2026-01-09] 삭제됨: 기존 기능 명세 -->
~~기존 기능: 로그인 시 자동 리다이렉트~~

### 새로운 내용 (수정 후)

<!-- [변경일: 2026-01-09] 변경 이유: 사용자 경험 개선을 위해 명시적 확인 단계 추가 -->
**변경된 기능**: 로그인 성공 시 확인 화면을 거쳐 홈 화면으로 이동
```

#### 수정 흐름
1. 개발 방향 변경 결정
2. **문서 업데이트** (`docs/` 폴더 내 관련 문서 수정)
   - 수정/삭제 부분 주석 처리 또는 취소선
   - 새로운 내용 추가
   - 변경 이력 기록
3. 변경된 문서 내용 확인 및 검토
4. **코드 변경 진행** (문서 변경 완료 후)
5. 로그 파일에 변경 사항 기록

#### 적용 대상 문서
- `docs/prd/` - PRD 문서
- `docs/wireframes/` - 화면 설계 문서
- `docs/user-flows/` - 유저 플로우 문서
- `docs/design-system/` - 디자인 시스템 문서
- `docs/task/` - To-Do List (필요 시)

## 3. 코드 작성 규칙

### 3.1 Flutter / Dart 규칙
- 사용자 규칙에 정의된 Flutter 개발 규칙 준수
- Riverpod, Freezed, Hooks 사용
- 문서, 로그, 커밋 메시지는 한국어를 기본으로 한다
- 코드 자체의 식별자/라이브러리 관례는 예외로 허용한다

### 3.2 코드 스타일
- `const` 생성자 사용
- `const` 위젯 최대한 활용
- 트레일링 콤마 사용
- 80자 줄 길이 제한

## 4. 테마 정책

### 4.1 라이트 테마 고정
- 사용자 스마트폰의 다크/라이트 환경과 무관하게 앱은 라이트 테마만 사용
- 앱의 라이트 테마는 독립적이며 스마트폰 환경에 영향을 받지 않음
- 다크 테마 관련 코드는 작성하지 않음

## 9. To-Do List 관리 규칙

### 9.1 To-Do List 폴더 구조
- 모든 To-Do List는 `docs/task/` 폴더에서 관리
- 주차별 To-Do List 파일로 구분 관리
- 파일명 형식: `N주차.md` (예: `1주차.md`, `2주차.md`)

### 9.2 To-Do List 작성 규칙
- 매번 작업 시 To-Do List를 업데이트
- 작업 시작 전 현재 주차 To-Do List를 확인
- 작업 완료 시 해당 항목을 체크하고 관련 로그 파일명을 기록

### 9.3 작업 항목 상태 관리
- 작업 전: `[ ]` (미완료)
- 작업 중: `[진행중]` (선택 사항, 진행 중임을 명시)
- 작업 완료: `[x]` (완료)

### 9.4 관련 로그 파일 결부 규칙 (필수)
- 작업 완료 시 반드시 해당 작업이 진행된 로그 파일명을 기록
- 작업 항목 뒤에 콜론(`:`)으로 구분하여 로그 파일명을 직접 기록
- 로그 파일명 형식: `YYMMDD.md` (예: `260108.md`)
- 여러 날에 걸쳐 완료된 작업의 경우 모든 관련 로그 파일명을 콤마로 구분하여 기록
  - 예: `260108.md, 260110.md`

### 9.5 To-Do List 업데이트 예시

#### 작업 전 상태
```markdown
### [ ] 로그인 기능
- [ ] 로그인 화면 UI 구현
- [ ] 아이디/비밀번호 로그인 연동
```

#### 작업 완료 후 상태
```markdown
### [x] 로그인 기능 : 260108.md
- [x] 로그인 화면 UI 구현 : 260108.md
- [x] 아이디/비밀번호 로그인 연동 : 260108.md
```

#### 여러 날에 걸친 작업 완료 예시
```markdown
### [x] 게시글 기능 개발 : 260108.md, 260110.md
- [x] 게시글 목록 화면 구현 : 260108.md, 260110.md
- [x] 게시글 작성 화면 구현 : 260109.md, 260110.md
```

### 9.6 주차별 진행 상황 업데이트
- 각 주차 파일 하단의 "진행 상황 요약" 섹션을 정기적으로 업데이트
- 완료 항목 수와 남은 항목 수를 정확히 기록
- 주차 마무리 시 체크리스트를 모두 완료

### 9.7 To-Do List와 로그 파일의 연결성
- To-Do List에서 완료된 작업은 반드시 관련 로그 파일명을 통해 개발 이력을 추적할 수 있어야 함
- 로그 파일에서 언급된 작업은 해당 주차 To-Do List에서 확인 가능해야 함
- 이를 통해 개발 이력의 일관성과 추적성을 보장
