# PRD - 비기능 요구 사항

## 8. 비기능 요구 사항 (Non-Functional Requirements)

본 섹션은 기능이 무엇을 하는지가 아니라,
해당 기능이 반드시 만족해야 하는 품질, 안정성, 운영 기준을 정의한다.

---

### 8.1 성능 (Performance)

#### 8.1.1 앱 초기 실행 시간
- 앱 실행 후 첫 화면 표시까지 3초 이내를 목표로 한다.
- 저사양 Android 기기에서도 동일 기준을 지향한다.
- 초기 로딩 단계에서는 `E:\app\jc_v4\image\slash`의 이미지 파일을 노출한다.

#### 8.1.2 화면 전환 반응성
- 화면 전환 및 주요 UI 반응은 지연 없이 즉각 반응하는 것처럼 느껴져야 한다.
- 게시글 목록, 회원 목록 진입 시 사용자가 로딩을 인지하지 않도록 한다.
- 불가피한 로딩이 발생하는 경우 명확한 로딩 상태를 표시한다.

#### 8.1.3 데이터 로딩 정책
- 회원 검색 및 목록 데이터는 100명 단위로 분할 로딩한다.
- 추가 데이터는 "더보기" 방식 없이, 스크롤 등 사용자 탐색에 따라 지속 호출하여 로딩한다.
- 최초 진입 시 전체 데이터를 한 번에 불러오지 않는다.

#### 8.1.4 이미지 처리
- 다음 기능이 가능하도록 구조를 설계한다.
  - 본문에 이미지를 삽입하며 글 작성
  - 게시글당 최대 5개의 이미지 업로드 (네이버 밴드 방식, 각각 500MB 이하)
  - 게시글 단위의 이미지 앨범 구성

---

### 8.2 안정성 (Stability & Reliability)

#### 8.2.1 앱 크래시 방지
- 일반적인 사용자 사용 시 앱이 종료되거나 멈추는 상황이 발생하지 않아야 한다.
- 예외 상황 발생 시 앱 전체가 종료되지 않고 정상 상태로 복귀해야 한다.

#### 8.2.2 네트워크 오류 대응
- 네트워크 연결이 불안정한 경우 다음을 반드시 제공한다.
  - 로딩 상태 표시
  - 재시도 가능 안내
- 네트워크 오류로 인해 사용자가 입력한 데이터가 손실되지 않아야 한다.
- 게시글 작성, 일정 등록 중 오류 발생 시 복구 가능해야 한다.

#### 8.2.3 알림 신뢰성
- 공지 및 일정 알림은 중복 발송되지 않아야 한다.
- 동일 이벤트에 대해 설정된 리마인더(푸시)는 정확한 시점에 1회만 발송된다.
- 알림 실패 시 재시도 또는 로그로 추적 가능해야 한다.

---

### 8.3 보안 (Security)

#### 8.3.1 접근 제어
- **가입 승인된 회원만 앱 사용 가능**
- **로그인하지 않은 사용자는 앱의 모든 기능에 접근 불가**
  - 로그인하지 않은 상태에서는 모든 화면 접근 차단
  - 하단 탭 네비게이션 숨김
  - 로그인 화면으로 자동 리다이렉트
- 회원 가입(계정 활성화)은 관리자 웹에서 승인된 사용자만 가능하다.
- 승인되지 않은 계정은 로그인 및 서비스 이용이 불가하다.
- 계정 상태(가입 대기, 승인, 비활성 등)는 서버 기준으로 관리된다.

#### 8.3.2 개인정보 보호
- 개인정보 보호 동의 없이는 회원 가입이 불가능하다.
- 프로필 정보는 필수 항목과 선택 항목이 명확히 구분되어야 한다.
- 선택 항목은 사용자가 입력한 경우에만 노출된다.

#### 8.3.3 관리자 권한 보호
- 관리자 웹은 최고 관리자(`manager_###`)만 접근 가능하다.
- 관리자 웹에서 비밀번호 생성 및 변경을 통해 관리자 계정 보안을 생성/유지/관리한다.
- 앱에서의 권한은 관리자 웹에서 부여해준 사용자 기준으로 관리된다.
  - 원칙적으로 직책이 있는 사용자에게만 권한을 부여한다.
- 권한이 부여되지 않은 사용자는 앱에서 권한이 필요한 기능에 접근할 수 없다.
- 권한 정보는 UI, API, 데이터 구조 상에서 추측 가능해서는 안 된다.
- 권한이 없는 사용자는 권한이 필요한 기능에 접근할 수 없어야 한다.

#### 8.3.4 데이터 접근 제한
- 회원 정보는 로그인한 사용자만 열람 가능하다.
- 외부 검색 엔진이나 외부 서비스에 노출되지 않아야 한다.
- 내부 데이터는 앱 외부 공유를 기본적으로 차단한다.

---

### 8.4 사용성 (Usability)

#### 8.4.1 IT 숙련도 낮은 사용자 고려
- 전역 회원 및 특우회 회원도 설명 없이 기본 기능을 사용할 수 있어야 한다.
- 불필요한 설정, 복잡한 단계, 중복 입력은 최소화한다.

#### 8.4.2 학습 비용 최소화
- 다음 핵심 기능은 앱 실행 후 2~3번의 터치 이내에 접근 가능해야 한다.
  - 공지 보기
  - 일정 확인
  - 회원 검색

#### 8.4.3 명확한 사용자 피드백
- 모든 주요 동작에 대해 성공 또는 실패 여부가 명확히 전달되어야 한다.
  - 회원 가입 요청
  - 가입 승인 (푸시 알림)
  - 게시글 작성
  - 일정 등록
- 실패 시 원인을 사용자에게 이해 가능한 수준으로 안내한다.

---

### 8.5 확장성 (Scalability)

#### 8.5.1 사용자 수 증가 대응
- 초기에는 영등포 JC 로컬 단위 사용을 전제로 한다.
- 회원 수 증가 시에도 성능 저하 없이 운영 가능해야 한다.

#### 8.5.2 조직 확장 고려
- 향후 다른 로컬 JC, 지구 JC, 한국 JCI 확장을 고려한다.
- 특정 로컬에 종속되는 구조적 제약이 없어야 한다.

#### 8.5.3 관리자 계정 확장
- 관리자 계정 규칙(manager_###)은 로컬 단위 확장 시에도 일관되게 적용 가능해야 한다.
- 관리자 권한은 소속 조직 단위로 분리 가능해야 한다.

---

### 8.6 운영 및 유지보수 (Operation & Maintenance)

#### 8.6.1 운영 정책 (초기/정상)
- 초기 버전에서는 관리자 권한 없이 운영 가능해야 한다.
- 정상 버전에서는 관리자 계정 기반 운영 정책으로 운영한다.

#### 8.6.2 데이터 변경 대응
- 다음 변경 사항 발생 시 기존 데이터와 충돌 없이 관리 가능해야 한다.
  - 회원 JC 직책 추가/변경
  - 소속 분과 추가/변경
  - 교육/연수 (신입/기존/임원/기타) 이력 및 기수, 년월 추가
  - 표창 
  - 국제대회 참가
  - 징계계
  - 소속 JC 변경

#### 8.6.3 로그 및 이력 추적
- 최소한의 수준에서 다음 이력은 추적 가능해야 한다.
  - 회원 승인 처리
  - 게시글 작성 및 삭제
  - 공지 등록
