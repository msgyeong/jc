## 1. 프로덕트 비전

### 단계별 확장 비전

```
[Phase 1] 영등포JC 단독 운영
    ↓ 안정화 + 실사용 검증
[Phase 2] 서울JC 23개 로컬JC 전파
    ↓ 멀티 테넌트 구조 적용
[Phase 3] 한국JC 전체 시스템으로 확장
```

### Phase 1: 영등포JC (현재 ~ MVP 출시)
- 대상: 영등포JC 회원 (약 50~100명)
- 목표: 핵심 기능 완성 + 실사용 검증
- 성공 기준: "이 앱만 보면 된다"는 인식 정착

### Phase 2: 서울JC 23개 로컬JC (MVP 출시 후 3~6개월)
- 대상: 서울JC 산하 23개 로컬JC
- 목표: 멀티 테넌트 구조로 각 로컬JC가 독립 운영
- 전제조건: 영등포JC에서 안정적 운영 실적 확보
- 핵심 과제: 테넌트 분리, 공통/독립 데이터 설계, 로컬 관리자 체계

### Phase 3: 한국JC 전체 (Phase 2 이후 6~12개월)
- 대상: 한국JCI 전체 (전국 로컬JC)
- 목표: 한국JC 공식 회원관리 시스템
- 핵심 과제: 대규모 인프라, 중앙 관리 기능, 로컬 간 교류 기능

## 2. 핵심 요구사항 (사장님 지시)

### R-01: 입회원서 디지털화

**배경**: 현재 JC 입회 시 워드로 입회원서를 별도 작성. 관리자가 수동으로 관리.

**요구사항**:
- 회원가입 시 입회원서의 모든 항목을 웹에서 입력
- 가입 완료 시 **회원관리카드가 자동 생성**
- 회원이 정보를 수정하면 회원관리카드도 **자동 업데이트**
- 관리자 웹에서 회원관리카드를 열람/출력/보완 입력 가능

**현재 상태**: 6단계 가입 폼이 구현되어 있으나, "회원관리카드" 형태의 통합 조회/출력 기능은 없음.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 가입 폼 항목 점검 | **기획자** | 실제 입회원서 양식과 비교하여 누락 항목 확인 |
| 회원관리카드 조회 화면 | **프론트엔드** | 관리자 웹에서 카드 형태로 회원 전체 정보 조회 |
| 회원관리카드 출력 기능 | **프론트엔드** | 인쇄/PDF용 레이아웃 |
| 회원 정보 수정 → 카드 자동 반영 | **백엔드** | 이미 동일 DB이므로 구조적으로 연동됨 |

### R-02: 참석 투표 시스템

**배경**: 이사회/월례회 등 주요 회의에 참석 여부를 사전에 파악해야 함.

**요구사항**:
- 공지사항 작성 시 **참석 투표 활성화** 옵션
- 회원은 공지 상세에서 **참석 / 불참 / 미정** 선택
- **이사회**: 성원(정족수) 충족 여부 실시간 표시
  - 예: "현재 참석 15명 / 성원 기준 20명 (75%)"
- **월례회**: 예상 참석 인원으로 비용 책정 지원
  - 예: "참석 예정 42명 × 식대 30,000원 = 1,260,000원"
- 투표 마감일 설정 가능
- 관리자: 참석 현황 목록 조회 + 미응답자 확인

**현재 상태**: 백엔드에 attendance API 일부 존재 (F-015). 프론트 UI 미완성. 성원/비용 계산 로직 없음.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 투표 유형 정의 (이사회/월례회/기타) | **기획자** | 유형별 성원 기준, 비용 계산 규칙 |
| 투표 데이터 모델 확장 | **백엔드** | vote_type, quorum, deadline, cost_per_person 필드 |
| 참석 투표 UI | **프론트엔드** | 공지 상세 내 투표 카드, 실시간 현황 |
| 성원/비용 계산 로직 | **백엔드** | 이사회: 성원 판정 / 월례회: 비용 산출 |
| 관리자 현황 조회 | **프론트엔드** + **백엔드** | 참석/불참/미정/미응답 목록 |

**참석 투표 데이터 구조 (안)**:
```
attendance_config: {
  vote_type: 'board_meeting' | 'monthly_meeting' | 'general',
  quorum_count: number | null,      // 성원 기준 인원 (이사회)
  quorum_basis: 'directors' | 'all', // 성원 기준 모수
  cost_per_person: number | null,    // 1인당 비용 (월례회)
  deadline: datetime | null,         // 투표 마감일
  is_active: boolean
}

attendance_vote: {
  user_id: number,
  notice_id: number,
  status: 'attend' | 'absent' | 'undecided',
  voted_at: timestamp
}
```

### R-03: 직함 연간 업데이트

**배경**: JC는 매년 직함(직책)이 변경됨. 회장, 부회장, 이사 등이 연 단위로 교체.

**요구사항**:
- 관리자가 회원의 직함을 업데이트하면 **즉시 전체 앱에 반영**
  - 회원 목록, 프로필, 게시글 작성자 정보 등 모든 곳
- 직함 변경 이력 관리 (연도별)
- 직함 변경 시 **권한 자동 반영** (PRD 기존 정책과 동일)
  - 예: 회장 → 일반회원으로 변경 시 공지 작성 권한 자동 해제

**현재 상태**: role 기반(super_admin/admin/member) 권한은 있으나, JC 직책 기반 권한은 미구현.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 직책-권한 매핑 테이블 정의 | **기획자** | 어떤 직책이 어떤 권한을 갖는지 |
| 직책 변경 API | **백엔드** | 관리자가 직책 업데이트 + 권한 자동 반영 |
| 직책 이력 테이블 | **백엔드** | position_history (user_id, position, year, ...) |
| 일괄 직책 변경 도구 | **백엔드** + **프론트엔드** | 연초 대량 변경 시 편의 기능 |

**직책-권한 매핑 (안)**:
```
| 직책         | 공지 작성 | 일정 등록 | 게시글 삭제 | 회원 관리 |
|-------------|----------|----------|-----------|----------|
| 회장         | ✅       | ✅       | ✅        | ❌ (웹만) |
| 상임부회장    | ✅       | ✅       | ✅        | ❌       |
| 내무부회장    | ✅       | ✅       | ✅        | ❌       |
| 사무국장     | ✅       | ✅       | ✅        | ❌       |
| 사무차장     | ✅       | ✅       | ❌        | ❌       |
| 이사         | ❌       | ❌       | ❌        | ❌       |
| 일반회원     | ❌       | ❌       | ❌        | ❌       |
```
> 관리자 웹에서 직책별 권한을 설정/변경 가능해야 함 (하드코딩 금지)

### R-04: 업종 검색

**배경**: "변호사 회원이 누구지?", "보험업 하는 분?" — 이름을 모르는 상태에서 업종으로 회원 탐색.

**요구사항**:
- 회원 검색 시 **업종** 필터/검색 지원
- 업종 카테고리 목록 제공 (선택형 + 직접 입력)
- 업종 검색 결과에서 해당 회원의 사업 정보 미리보기

**현재 상태**: 기본 검색(이름/이메일/전화/주소/회사명)만 구현. 업종 필드는 DB에 있으나 검색 미지원.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 업종 카테고리 목록 정의 | **기획자** | 법률/회계/금융/의료/요식/건설/IT/제조/유통/서비스 등 |
| 업종 검색 API 확장 | **백엔드** | GET /api/members/search?industry=법률 |
| 검색 UI 필터 추가 | **프론트엔드** | 업종 드롭다운 또는 태그 필터 |

### R-05: 배너 광고 (회원 사업장 홍보)

**배경**: 회원 사업장을 홈 배너로 홍보. 회원이 신청하면 관리자가 승인하여 노출.

**요구사항**:
- 회원이 **배너 광고 신청** 가능 (사업장 정보 + 이미지 + 링크)
- 관리자가 **신청 승인/거절** 후 배너 노출
- 홈 화면 배너 슬라이더에 노출 (기존 배너 기획과 통합)
- 노출 기간 설정 (시작일/종료일)
- 배너 클릭 → 사업장 홈페이지 또는 프로필 상세로 이동

**현재 상태**: 배너 UI 구조만 존재. 관리 기능 없음.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 배너 신청/승인 프로세스 정의 | **기획자** | 신청 폼, 승인 절차, 노출 규칙 |
| 배너 신청 API + 관리 API | **백엔드** | 회원 신청 + 관리자 승인/거절 + CRUD |
| 배너 신청 UI (프로필 또는 별도 화면) | **프론트엔드** | 신청 폼 |
| 배너 관리 UI (관리자) | **프론트엔드** | 신청 목록 + 승인/거절 |

---

### R-06: 프로필 자기 PR (사업 정보 노출)

**배경**: 회원이 본인 사업 정보를 원하는 만큼 입력하여 다른 회원에게 노출. 회원 간 사업 시너지 창출.

**요구사항**:
- 프로필에 **"사업 PR" 섹션** 추가
- 자유 형식으로 사업 정보 입력:
  - 사업장명
  - 업종/분야
  - 사업 소개 (자유 텍스트, 최대 500자)
  - 대표 이미지 (최대 3장)
  - 홈페이지/SNS 링크
  - 연락처 (사업용)
  - 주소/위치
- 다른 회원이 프로필 조회 시 사업 PR 섹션 노출
- 회원 검색 결과에서 사업 정보 미리보기
- **본인만 수정 가능** (관리자 수정 대상 아님)

**현재 상태**: 프로필에 기본 직장 정보(회사명, 직책)만 존재. 자기 PR 섹션 없음.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 사업 PR 데이터 모델 | **백엔드** | business_pr 테이블 또는 users JSONB 확장 |
| 사업 PR 입력/수정 UI | **프론트엔드** | 프로필 편집 내 PR 섹션 |
| 사업 PR 노출 UI | **프론트엔드** | 프로필 상세 + 검색 결과 미리보기 |
| 사업 PR API | **백엔드** | CRUD 엔드포인트 |

### R-07: 공지-일정 연동

**배경**: 공지사항에 일정 정보가 포함되면 캘린더에 자동 반영. 수동 중복 등록 방지.

**요구사항**:
- 공지 작성 시 **일정 첨부 옵션** 제공
  - 일정 제목, 날짜, 시간, 장소, 카테고리
- 공지 저장 시 **일정 테이블에 자동 생성**
- 캘린더에서 해당 일정 선택 시 **연결된 공지로 이동** 가능
- 공지 수정 시 연결된 일정도 **동기화**
- 공지 삭제 시 연결된 일정 처리 방식 결정 필요 (삭제 or 유지)

**현재 상태**: 공지와 일정이 완전히 분리된 독립 기능. 연동 없음.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 연동 데이터 구조 설계 | **기획자** + **백엔드** | post_id ↔ schedule_id 연결 |
| 공지 작성 시 일정 입력 UI | **프론트엔드** | 토글 + 일정 입력 필드 |
| 연동 API | **백엔드** | 공지 CRUD 시 일정 자동 생성/수정/삭제 |
| 캘린더 → 공지 이동 | **프론트엔드** | 일정 상세에서 "연결된 공지 보기" |

## 3. 확장성 설계: 멀티 테넌트 구조

### 3-1. 현재 구조 (싱글 테넌트)
```
[영등포JC 앱] → [단일 DB] → [단일 서버]
```
- 모든 데이터가 하나의 DB에 존재
- organization 개념 없음

### 3-2. Phase 2 목표 구조 (멀티 테넌트)
```
[로컬JC 앱] → [공유 API 서버] → [테넌트별 데이터 분리]

┌─────────────────────────────────┐
│         공유 인프라              │
│  ┌─────────────────────────┐   │
│  │   API 서버 (공용)        │   │
│  │   인증 서버 (공용)       │   │
│  └─────────────────────────┘   │
│                                 │
│  ┌──────┐ ┌──────┐ ┌──────┐   │
│  │영등포│ │ 강남 │ │ 마포 │...│  ← 23개 로컬JC
│  │ JC  │ │ JC  │ │ JC  │   │
│  └──────┘ └──────┘ └──────┘   │
│  (테넌트별 독립 데이터)          │
└─────────────────────────────────┘
```

### 3-3. 멀티 테넌트 전환을 위한 사전 준비 (Phase 1에서 고려)

지금 당장 멀티 테넌트를 구현하지는 않지만, **확장 시 마이그레이션 비용을 줄이기 위해** 아래 사항을 고려한다:

| 항목 | 현재 조치 | Phase 2 전환 시 |
|------|----------|----------------|
| **organization_id 필드** | 주요 테이블에 `org_id` 예비 컬럼 추가 고려 | 테넌트 분리 키로 활용 |
| **직책/권한 설정** | 하드코딩 금지, DB 설정 테이블 사용 | 로컬JC별 독립 권한 체계 |
| **배너/홍보** | 관리자 등록 방식 | 로컬JC별 독립 배너 |
| **회원 데이터** | 영등포JC 단일 | 로컬JC별 회원 분리 + 교차 검색 (Phase 3) |
| **URL/접근** | 단일 도메인 | 서브도메인 (ydp.jcapp.kr, gangnam.jcapp.kr) 또는 앱 내 선택 |

### 3-4. Phase 3 추가 고려사항

- **중앙 관리 대시보드**: 한국JC 본부에서 전체 로컬JC 현황 조회
- **로컬 간 교류**: 타 로컬JC 회원 프로필 조회 (제한적)
- **통합 행사**: 한국JC 전체 행사 일정 공유
- **통합 회원 디렉토리**: 전국 회원 업종/지역 검색

### R-08: 프로필 바로 통화

**배경**: 회원 프로필에서 바로 전화를 걸 수 있어야 함. 별도로 번호를 복사/저장하는 번거로움 제거.

**요구사항**:
- 회원 프로필 상세에서 전화번호 옆 **전화 아이콘** 표시
- 아이콘 클릭 시 `tel:` 링크로 **바로 통화 연결**
- 전화번호가 비공개 설정인 회원은 아이콘 미표시
- 사업용 연락처 (R-06 사업 PR)도 동일하게 바로 통화 지원

**현재 상태**: 프로필에 전화번호 텍스트만 표시. 클릭 동작 없음.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 전화 아이콘 + tel: 링크 | **프론트엔드** | 프로필 상세 화면에 아이콘 추가 |
| 공개/비공개 연동 | **프론트엔드** | 비공개 설정 시 전화 아이콘 숨김 |

### R-09: 즐겨찾기 기능

**배경**: 자주 연락하는 회원을 즐겨찾기하여 빠르게 접근. 매번 검색할 필요 없이 바로 연결.

**요구사항**:
- 회원 프로필에서 **별표(즐겨찾기) 토글** 버튼
- 회원 목록 화면에 **"즐겨찾기" 탭 또는 필터** 추가
- 즐겨찾기한 회원은 상단 우선 표시 (옵션)
- 즐겨찾기 데이터는 **사용자별 로컬** (다른 회원에게 노출되지 않음)

**현재 상태**: 미구현.

**구현 필요 사항**:
| 항목 | 담당 | 내용 |
|------|------|------|
| 즐겨찾기 데이터 모델 | **백엔드** | favorites 테이블 (user_id, target_user_id) |
| 즐겨찾기 API | **백엔드** | POST/DELETE /api/members/:id/favorite, GET /api/members/favorites |
| 즐겨찾기 토글 UI | **프론트엔드** | 프로필 상세 + 회원 목록에서 별표 아이콘 |
| 즐겨찾기 필터/탭 | **프론트엔드** | 회원 목록 상단에 즐겨찾기 필터 |

**데이터 구조**:
```
favorites: {
  id: serial,
  user_id: integer,          // 즐겨찾기한 사용자
  target_user_id: integer,   // 즐겨찾기된 회원
  created_at: timestamp
}
// UNIQUE(user_id, target_user_id)
```

### R-10: 관리자 웹 전면 재개발

**배경**: 기존 admin-web 문서/코드를 전부 폐기하고, 사장님이 제공한 UI 명세서 기반으로 새로 개발.

**요구사항**:

- 상세 화면별 기능: `Docs/admin-web/` 문서 참조 (AW-001 ~ AW-009)

  - Primary #2563EB, Danger #DC2626
  - 카드 Radius 16px, Content Padding 32px

**구현 필요 사항**:
